# Codex 链路 v0.3 实施 Roadmap

## 1. 状态

Planned

这份文档用于描述 `agent-harness` 在 `Codex` 宿主下，从“当前最小闭环已实现”走向“符合 v0.3 设计目标”的后续实施路径。

它不替代：

- [Agent Harness 设计文档 v0.3](docs/2026-04-03-agent-harness-design-v0.3.md)
- [Codex 自动 Intake 设计稿 v0.1](docs/2026-04-03-codex-auto-intake-design-v0.1.md)
- [Codex Hooks 开发流接入说明](docs/2026-04-03-codex-hooks-workflow-v0.1.md)

它回答的是一个更直接的问题：

`Codex` 链路从当前状态走到 `v0.3` 完整实现，还需要做哪些工作，优先级如何排。

## 2. 当前判断

当前 `Codex` 链路已经具备：

- `SessionStart` 恢复 active task
- `UserPromptSubmit` 自动 intake
- `task / state / verify / report / gate / audit` 最小闭环
- `task-core` 的可解释分类结果与最小回归样本

因此，当前状态不是“未开始”，而是：

`最小闭环已实现，但尚未达到 v0.3 完整实现`

## 3. 完整实现的判定标准

当以下条件成立时，可认为 `Codex` 链路达到 `v0.3` 完整实现：

1. `Task Draft -> Confirmed Contract` 的闭合链路清晰可用
2. 关键 evidence 能自动采集，不再高度依赖手工补录
3. `Codex` 不只具备最小 intake，而是具备更完整的 lifecycle 接入
4. `delivery_policy / output_policy` 不只是配置存在，而是被工作流真实消费
5. `CHANGELOG / design note / ADR / commit-ready` 已进入交付收口流程
6. `Codex` 端拥有可复现的端到端回归验证样本

## 4. Roadmap

## 4.1 P0 必做

目标：

把 `Codex` 从“最小可用”推进到“任务闭环稳定可用”。

### P0.1 Confirmed Contract 闭合链路

需要完成：

- 明确 `Task Draft` 何时升级为 `Confirmed Contract`
- 定义 clarify 之后的字段回写规则
- 定义哪些字段闭合后才能进入稳定执行态

完成标准：

- 不再只有 `Task Draft`
- 合同闭合时机、写入位置和后续消费点明确

### P0.2 自动 Evidence 采集

需要完成：

- 命令结果自动进入 `evidence`
- 验证结果自动进入 `evidence`
- override / manual confirmation 自动进入 `audit`

完成标准：

- 常见开发路径下不需要频繁手工 `state update --evidence`

### P0.3 Codex Hook 主链路稳定化

需要完成：

- 明确 `SessionStart` / `UserPromptSubmit` 的降级策略
- 明确 trusted / untrusted project 的行为差异
- 固化 hook 失败时的用户可见提示

完成标准：

- hook 失效时不会 silently fail
- 用户知道当前是自动模式还是降级模式

### P0.4 Codex E2E 回归集

至少覆盖：

- 新任务自动 intake
- follow-up prompt 不误切任务
- 歧义 prompt clarify / block
- trusted / untrusted project 差异
- hook 失败后的降级路径

完成标准：

- `Codex` 主链路有稳定回归样本，不只靠临时手工验证

## 4.2 P1 应做

目标：

把 `v0.3` 中已经定义的 project policy 真正接进工作流。

### P1.1 Output Policy 消费完整化

需要完成：

- `output_policy.changelog`
- `output_policy.design_note`
- `output_policy.adr`

至少先实现：

- 判定
- 提示
- 校验

完成标准：

- changelog / design note / ADR 不再只是文档约束，而是流程约束

### P1.2 Delivery Policy 落地

需要完成：

- 定义 `commit-ready`
- 定义 `delivery_policy.commit`
- 定义 `delivery_policy.push` 的人工边界与 readiness 提示

原则保持：

- `commit` 只在显式请求时执行
- `push` 只在显式请求后由人工执行，不作为 skill 默认能力

完成标准：

- “任务完成”与“仓库已可提交”被明确区分

### P1.3 复杂任务工件沉淀

需要完成：

- 判断何时必须写 `design note`
- 判断何时必须写 `ADR`
- 定义最小模板或最小字段

完成标准：

- 复杂任务收口时不再完全依赖人工习惯

## 4.3 P2 可做

目标：

从“工作流完整”推进到“宿主能力更强、更稳”。

### P2.1 更强的 Lifecycle Hooks

可以评估：

- `PreToolUse`
- `PostToolUse`
- completion 前的宿主校验入口

目标不是为了 hooks 更多，而是为了让 `Codex` 侧状态、证据和门禁更自然地接入开发流。

### P2.2 更强 Gate

需要增强：

- 更细粒度的 scope 约束
- 更强的高风险动作阻断
- 更贴近宿主实际行为的门禁判断

### P2.3 Task Core 持续进化

继续按当前原则推进：

- 先补 fixture
- 再改规则
- 没有真实误判样本时先观测

### P2.4 Skill 化交付动作

可逐步承接：

- commit skill
- changelog 更新辅助
- design note / ADR 草稿辅助

## 5. 推荐实施顺序

建议按以下顺序推进：

1. `P0.1 Confirmed Contract`
2. `P0.2 自动 Evidence`
3. `P0.3 Hook 稳定化`
4. `P0.4 Codex E2E`
5. `P1.1 Output Policy`
6. `P1.2 Delivery Policy`
7. `P1.3 Design Note / ADR`
8. `P2` 宿主强化与 skill 化

## 6. 当前停点建议

如果要在当前节点开始继续开发，最值得优先做的是：

1. `Confirmed Contract` 闭合链路
2. 自动 evidence 采集

原因：

- 这两项决定了 `Codex` 链路是否真正从“最小可用”进入“稳定可用”
- 如果这两项不补，后面的 changelog、commit-ready、design note 都会建立在不够稳定的主链路之上

## 7. 结论

对 `Codex` 链路的当前判断应统一为：

- 不是“未实现”
- 也不是“已经完全完成”
- 而是“最小闭环已实现，距离 v0.3 完整目标还差 P0/P1/P2 三层工作”
