# Codex Hook 可见性规范 v0.1

## 目标

在保留 `Codex hooks` 作为 `agent-harness` agent-native 核心接入层的前提下，约束用户前台可见信息，避免把内部控制流误暴露为产品交互。

本规范回答两个问题：

- 哪些 hook 能力必须保留
- 哪些信息允许被用户看到

## 设计结论

`hook` 属于控制面，不属于对话面。

- 控制面负责：状态恢复、任务 intake、执行门禁、evidence 记录、override 审计
- 对话面负责：给用户结论，或在必要时向用户请求决策

默认情况下，用户不应因为 hook 正常运行而被迫看到额外说明。用户前台只应出现两类高价值信息：

- `结论`
- `决策请求`

## 术语

### 控制面

指为保障 agent 按协议运行而存在的内部运行时能力，包括：

- `SessionStart`
- `UserPromptSubmit`
- `PreToolUse`
- `PostToolUse`
- task state / audit / verify / report 等状态机能力

### 对话面

指用户实际感知到的交互输出，包括：

- 模型结论
- 单个最高价值澄清问题
- 高风险确认请求
- 路径分叉下的用户决策请求

### 可见性

指某一信息是否应直接出现在用户前台，而不是仅存在于状态、审计或内部上下文中。

## 基本原则

### 原则 1：控制面默认静默

正常运行的 hook 应视为基础设施，不应默认向用户解释自身存在。

允许静默的典型信息：

- active task 已恢复
- 当前输入已被 intake 为 follow-up / new task
- override 已被记录
- evidence 已被追加
- 普通 continue 决策已完成

### 原则 2：只有阻断和决策可以前置

只有当用户不参与就无法继续时，控制面信息才允许穿透到对话面。

允许前台显示的情况：

- 当前请求缺少完成判据，无法继续
- scope 不清且可能越过保护边界
- 命中高风险范围，需要用户确认
- 存在多个成本差异明显的实现路径，需要用户拍板

不应前台显示的情况：

- 单纯的 task 归属判断
- 自动恢复 active task
- 正常的 fail-open continue
- 审计记录成功
- hook 自身开始执行或执行结束

### 原则 3：用户只为决策负责

用户的主要职责是做业务或风险决策，而不是阅读 agent 内部运行过程。

因此默认交互应收敛为：

1. 模型先给出结论或建议路径
2. 如存在阻断，只提出一个最高价值问题
3. 如存在高成本分叉，只请求一次明确决策

## 当前 hooks 的职责边界

### SessionStart

职责：

- 恢复 active task 摘要
- 为当前会话提供最小任务上下文

可见性要求：

- 默认静默
- 不得为了“解释已恢复什么”而主动占用前台输出

例外：

- 恢复失败且影响后续任务归属判断时，可降级为手动 fallback 提示

### UserPromptSubmit

职责：

- 判断当前输入是续写当前任务、创建新任务，还是需要澄清
- 识别用户显式 override / manual confirmation
- 在必要时执行 block

可见性要求：

- `continue` 应静默
- `override` 记录应静默
- 只有 `needs_clarification` 或 `require_confirmation` 才允许升级到前台

对话面约束：

- 一次只暴露一个最高价值阻断问题
- 不向用户展示完整 intake 推断过程

### PreToolUse

职责：

- 在工具调用前做确定性门禁检查
- 阻止越界写入、未澄清执行或高风险未确认执行

可见性要求：

- 允许阻断
- 阻断文案应说明“为什么当前不能继续”
- 不应暴露实现细节或完整内部规则树

### PostToolUse

职责：

- 自动记录命令或测试 evidence

可见性要求：

- 默认静默
- evidence 成功记录不应成为用户可见提示
- 仅在记录失败且会影响最终完成门禁时，才允许在最终答复中说明

## 前台输出规则

### 允许的前台输出

- 结论
- 单个阻断问题
- 高风险确认请求
- 路径分叉下的决策请求
- 与最终交付直接相关的验证结果

### 禁止的前台输出

- “正在运行某个 hook”式过程提示
- 完整 intake 字段推断
- task state 内部迁移细节
- audit 追加成功提示
- evidence 自动记录成功提示
- 与继续执行无关的内部 reasoning

## 宿主约束说明

需要区分两类可见信息：

1. `harness` 自身输出的业务信息
2. `Codex` 宿主 UI 对 hook 生命周期的固定提示

当前仓库能控制的是前者，未必能完全控制后者。

因此：

- `harness` 侧必须做到业务层默认静默
- 若宿主仍显示 `Running SessionStart hook`、`Running UserPromptSubmit hook` 等提示，应视为宿主展示层约束，而不是业务层文案设计失败

后续如宿主支持 silent hooks、low-visibility hooks 或类似特性，应优先切换到该模式。

## 实现约束

实现或评审 Codex hook 时，必须满足以下约束：

- `continue` 路径不得输出用户解释性文案
- `block` 路径只能输出继续执行所必需的信息
- 不得把完整 task context、reasoning note 或推断链路直接暴露到前台
- fail-open 降级文案应仅在确实影响使用时出现
- 不得为了便于调试而默认增加前台噪音

## 验收标准

满足以下条件，才可视为符合本规范：

- 在正常 `continue` 路径下，用户前台只看到模型回复，不看到额外业务层 hook 文案
- 需要澄清时，前台只出现一个最高价值问题
- 需要确认时，前台只出现一次明确确认请求
- `PreToolUse` 阻断时，用户能理解阻断原因，但看不到内部实现细节
- `PostToolUse` 正常记录 evidence 时，前台无额外提示

## 非目标

本规范当前不解决以下问题：

- `Codex` 宿主是否支持彻底隐藏 hook 生命周期提示
- 非 `Codex` 宿主的可见性统一策略
- 更复杂的多任务可视化或后台任务面板设计

## 建议的后续动作

- 将本规范作为 `Codex` 接入层评审基线
- 后续新增 hook 时，先判断其是否属于控制面能力
- 若宿主支持更细的 hook 可见性开关，优先将 `SessionStart` / `UserPromptSubmit` / `PostToolUse` 切到静默模式
- 如需面向用户展示系统状态，应单独设计“状态面板”，不要复用主对话流
